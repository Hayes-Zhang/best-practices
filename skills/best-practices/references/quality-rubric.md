# 最佳实践质量评估标准

对搜索到的每条最佳实践，从以下四个维度评估。每个维度 0-3 分。

---

## 维度 1：权威性（Authority）

**谁说的？来源有多可信？**

| 分数 | 标准 | 举例 |
|------|------|------|
| 3 | 官方文档或标准制定者 | React 官方文档推荐、RFC 规范、AWS Well-Architected |
| 2 | 领域专家或头部公司工程博客 | Dan Abramov 的文章、Netflix Tech Blog |
| 1 | 社区高票内容 | Stack Overflow 100+ upvotes、知乎 500+ 赞 |
| 0 | 来源不明或个人博客无背书 | 随机 CSDN 文章、无引用的 Medium 帖子 |

---

## 维度 2：共识度（Consensus）

**多少独立来源持相同观点？**

| 分数 | 标准 |
|------|------|
| 3 | 3+ 个独立权威来源一致推荐 |
| 2 | 2 个独立来源一致，且无反面证据 |
| 1 | 仅 1 个来源，或有部分争议 |
| 0 | 仅 AI 生成内容，或来源之间严重矛盾 |

**注意**：多个来源引用同一原始出处只算 1 个来源。要看独立性。

---

## 维度 3：可操作性（Actionability）

**看完能直接照着做吗？**

| 分数 | 标准 | 举例 |
|------|------|------|
| 3 | 有具体步骤 + 代码示例 + 配置参考 | "第一步安装 X，第二步配置 Y，示例代码如下…" |
| 2 | 有明确方向但缺少细节 | "推荐使用 SSR + ISR 混合策略" |
| 1 | 有原则但太抽象 | "要注意性能优化" |
| 0 | 纯理论或观点性内容 | "这个领域很重要" |

---

## 维度 4：时效性（Recency）

**信息是否还有效？**

| 分数 | 标准 |
|------|------|
| 3 | 6 个月内发布/更新，且目标技术未发生重大变化 |
| 2 | 1 年内发布，核心建议仍适用 |
| 1 | 1-3 年前发布，需要验证是否过时 |
| 0 | 3 年以上或目标技术已大版本更新（如 React Class → Hooks） |

**例外**：经典原则（如 SOLID、12-Factor App）不受时间限制，时效性默认 3 分。

---

## 综合评分与决策

```
综合分 = 权威性 × 0.3 + 共识度 × 0.3 + 可操作性 × 0.2 + 时效性 × 0.2
```

| 综合分 | 判断 | 行动 |
|--------|------|------|
| ≥ 2.0 且共识度 ≥ 2 | ✅ Found（成熟实践） | 直接输出建议 + 来源 |
| ≥ 1.0 或有部分共识 | 🔧 Synthesized（综合实践） | 综合多来源后输出，明确标注"综合多方" |
| < 1.0 | 💡 Deduced（原创推导） | 切换到 first-principles 模式：基于核心约束推导一套方案，展示推导逻辑 + 验证建议，明确标注"推导而非找到"。**不要拒绝回答**。可在结尾附"想要更深入调研？跑 /bp"作为可选升级路径 |

---

## 红旗信号（直接降级）

遇到以下情况，无论单项分数多高，整体降一级：

- **利益冲突**：来源在推销自己的产品/服务
- **过度简化**：复杂问题只给了一个"银弹"方案
- **缺少 trade-off**：只说好处不说限制
- **AI 生成嫌疑**：内容高度通用、缺乏具体经验
