# Best Practices for Claude Code

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
[![Version](https://img.shields.io/badge/version-2.2.0-blue.svg)](CHANGELOG.md)
[![Claude Code](https://img.shields.io/badge/Claude_Code-Plugin-orange)](https://docs.claude.com/en/docs/claude-code/plugins)

**给 Claude Code 一双生态感知的眼睛。** 在动手回答或写代码前，插件先帮你查：
你的场景是不是**已经有人想过/做过成熟方案**？

- ✅ **有直接答案** → 直接给（附来源和 trade-off）
- 🧩 **只有邻近场景有成熟方案** → 插件帮你借鉴（提炼共性原则 + 为你的场景定制）
- 💡 **连邻近场景也没有** → 退到第一性原理推导，展示完整推导链

每一级都**诚实标注**，让你知道这个答案背后是什么证据。
适用于**任何**"我该怎么做 X？"的问题——技术、产品、生活技能都行。

## 为什么需要它

AI 在"how do I X"类问题上最常见的失败模式，不是幻觉——是**重复造轮子**。

轮子有很多形状。有时候是 SaaS 集成（多 LLM 接入？**OpenRouter**，不是一个个
接 API）。有时候是成熟开源库（AI 应用 UI 流式输出？**Vercel AI SDK**，不是
自己撸 SSE）。有时候是工程模式（离线优先同步？**CRDT + 操作日志**，不是
last-write-wins）。有时候是流程方法（新员工 1on1？**First Round / Lara Hogan
模板**，不是即兴发挥）。有时候是设计原则（组件库 API？**Atomic Design + Design
Tokens**，不是"先这么写着"）。

每种情况里，AI 都**知道怎么做**——它能手撸 SaaS、能写库代码、能推导模式、能
起草 playbook。它不知道的是**这个领域已经收敛到了什么共识**——因为成熟方案
（无论是 SaaS、开源库、工程模式、还是 playbook）在训练数据里的曝光，都远远
不如"手写教程"。

这个插件就是来补这个口子的。AI 动手干之前，先问一句：**这件事这个领域有没有
已经收敛的成熟方案？** 有就用现成的——无论它是 SaaS、库、模式、playbook、还是
设计原则。没有但邻近场景有，就借鉴邻近方案再定制。**什么都没有，才退到第一性
原理推导**——而且要明说自己是在推。

简单粗暴的"每次都搜网"也有自己的失败模式：SEO 文章、AI 灌水、社区互相矛盾
的帖子，会以更"权威"的样子把模型带偏到另一个错方向。所以这个插件也会**评估**
搜到的资料质量（5 个红旗检测），再决定是否信任。

当你问到你不熟悉的问题，AI 会：

1. **搜索**这个话题已有的最佳实践
2. **评估**找到的资料质量（5 个红旗 + 反向思考）
3. **挑选正确的 tier**——✅ Found / 🔧 Synthesized / 🧩 Adapted / 💡 Deduced
4. **🧩 Adapted（关键）**：当找不到直接答案但有邻近成熟方案时，执行三段式
   （邻近方案 → 共性原则 → 当前场景定制）——绝不把借鉴来的建议伪装成"业界共识"
5. **💡 Deduced（最后兜底）**：只有完全找不到邻近借鉴时，才退到第一性原理推导，
   展示完整推导链

适用于所有 *"我该怎么做 X？"* 类问题——**不限于技术**。

## 不只是技术问题

"最佳实践"有很多形状，同样的"先看这个领域有没有共识"逻辑，对所有形状都成立。

**技术 — SaaS / 集成（生态有托管方案）：**
> *"我要给 app 加多 LLM 接入"* → **OpenRouter**，不是一个个接 API
> *"我要加用户认证（Next.js）"* → **Clerk / Auth.js**，不是手撸 JWT
> *"我要加文件上传"* → **Uploadthing / R2**，不是手撸 multipart
> *"我要加实时通信"* → **Pusher / Liveblocks**，不是自建 WebSocket

**技术 — 库 / 框架（社区已收敛到某个开源方案）：**
> *"AI 应用怎么把回复流式输出到 UI？"* → **Vercel AI SDK**，不是自己撸 SSE
> *"这个 React 项目怎么管状态？"* → **Zustand / Jotai**，不是 Context+useReducer
> *"怎么做类型安全的数据库访问？"* → **Drizzle / Prisma**，不是手写 SQL builder

**技术 — 工程模式（领域已经收敛的方法）：**
> *"多设备离线优先同步怎么设计？"* → **CRDT + 操作日志**，不是 last-write-wins
> *"长任务怎么可靠跑完？"* → **Outbox 模式 + 幂等 worker**，不是"重试就好"
> *"API 怎么做向后兼容的迭代？"* → **Expand-Contract 迁移**，不是分版本 fork

**设计 — 系统 / 原则（成熟设计语言）：**
> *"组件库怎么设计 API？"* → **Atomic Design + Design Tokens**，不是临时定
> *"app 怎么默认就无障碍？"* → **WCAG + ARIA 模式**，不是"先做完功能再加 label"

**流程 / 沟通（成熟 playbook）：**
> *"新员工 1on1 怎么开？"* → **First Round / Lara Hogan 模板**
> *"OKR 怎么设才不流于形式？"* → **Christina Wodtke 的 Radical Focus**
> *"PR description 怎么写别人才愿意看？"* → **Conventional commit + 正文规范**

**生活技能（开发圈外也有成熟实践）：**
> *"挑食的小孩吃饭怎么办？"* → **Division of Responsibility（Ellyn Satter 育儿法）**
> *"远程团队异步 standup 怎么开？"* → **GitLab handbook 模式**

模式很简单：**只要这件事别人已经想过、而你脑子里还没有这个答案**，就适用。
直接答案存在（无论以哪种形式——SaaS、库、模式、playbook、原则）就直接给；
只有邻近场景有成熟方案，就借鉴邻近的并为你的场景定制；都没有才推导——
并且明确告诉你是哪种。

## 快速开始

### 安装

```bash
# 添加 marketplace
/plugin marketplace add Hayes-Zhang/best-practices

# 安装插件
/plugin install best-practices@best-practices
```

### 试一下

在新对话里问任何 how-to 问题。下面这些都应该触发：

> *"我要给 app 加多 LLM 接入怎么做？"*（技术 → 大概率 ✅ Found：OpenRouter SaaS）
> *"中等规模的 React 项目怎么管状态？"*（技术 → 大概率 ✅ Found：Zustand 库）
> *"API 怎么做向后兼容的迭代？"*（技术 → 大概率 ✅ Found：Expand-Contract 模式）
> *"团队的 oncall 排班一般怎么安排比较合理？"*（流程 playbook）
> *"冷启动邮件怎么写对方才会回？"*（沟通）

你会看到回答末尾带其中一个 tier 标签：
- **✅ Found（找到）** — 同场景多个独立来源一致推荐
- **🔧 Synthesized（综合）** — 同场景多个碎片来源拼出来的
- **🧩 Adapted（定制）** — 同场景没有直接答案，但邻近场景有成熟方案 →
  执行三段式（邻近方案 → 共性原则 → 当前场景定制）
- **💡 Deduced（推导）** — 同场景**和邻近场景**都没有可借鉴的方案 →
  基于第一性原理推导，展示完整推导链（最后兜底）

需要更深入的多角度调研：

```
/bp "你的话题，技术的或非技术的都行"
```

更多真实输出见 **[examples/](examples/)**。

## 触发清单

插件设计为自动激活，但只在真的有价值时触发。

| ✅ 触发 | ❌ 不触发 |
|---|---|
| how-to 类问题（*"怎么做 X？"*） | 已经在执行（*"帮我把这段翻译成中文"*） |
| 选型对比（*"X vs Y"*） | 基础概念（*"什么是闭包"*） |
| 明示时效（*"最新的"*、*"2026 的"*） | 数学 / 经典算法 |
| 明示版本号（*"React 19"*、*"Tailwind v4"*） | 修具体代码 bug |
| 变化快领域（AI、前端、云服务、设计系统） | 纯创意任务（写诗、起名字） |
| 用户不熟悉的领域 | 用户已经选定方案 |
| 明确关键词（*"最佳实践"*、*"推荐"*） | 元对话（*"你能做什么"*） |
| 非技术 how-to（1:1、招聘、育儿、做菜等） | |

**多搜 30 秒的代价很小。confidently 给一个错答案的代价很大。不确定就搜。**

## 工作原理

### 三层判断

1. **触发检测**——这个问题值得搜吗？
2. **反向思考评估**（默默进行，回答前完成）：
   - 这些来源真的独立吗？还是都引用同一个原始出处？
   - 我搜过反面证据吗？
   - 这些建议有 trade-off 吗？还是所有人都在吹？
   - 信息有多新？这个领域变化多快？
3. **四级分级回答**（每一级的诚实标签都对应不同的证据类型）：
   - **✅ Found** — 多个独立来源在同场景一致 → 直接给建议
   - **🔧 Synthesized** — 同场景碎片化共识 → 给建议但标注"综合多方"
   - **🧩 Adapted** — 同场景无直接答案，但邻近场景有成熟方案 →
     执行三段式（邻近方案 → 共性原则 → 当前场景定制），明确标注借鉴来源和跨场景迁移风险
   - **💡 Deduced** — 同场景**和邻近场景**都无可借鉴 →
     基于第一性原理推导 + 完整推导链 + 明确标注（最后兜底）

### 五个红旗

插件主动检测并明示：

- 🚩 **利益冲突**——来源在推销自己的产品
- 🚩 **SEO 文章特征**——标题党、内容空泛、没有具体示例
- 🚩 **AI 生成嫌疑**——高度通用、缺少具体经验
- 🚩 **银弹思维**——复杂问题只给一个方案，没有 trade-off
- 🚩 **独立性假象**——多个来源其实都引用同一个原始出处

## Trust & Safety

整个插件围绕一个原则：**诚实的来源标注 > 自信的瞎猜，诚实的邻近场景借鉴 >
编造的"业界共识"，诚实的第一性原理推导 > 自信的瞎猜——但前提是每个输出都
清晰标 tier。**

- 每个引用的 URL 都是真实的。URL 不存在就不编造
- 找不到直接最佳实践时，插件**优先**找邻近场景的成熟方案做透明借鉴
  （🧩 Adapted），明确标注借鉴自哪个邻近场景，并提示跨场景迁移的风险
- 连邻近场景也找不到时，插件**不会拒绝回答**——它会切到第一性原理推导
  （💡 Deduced），展示完整推导链，并明确标注"推导"而非"找到"
- AI 推导的建议（无论是 🧩 Adapted 还是 💡 Deduced）绝不会伪装成
  ✅ Found 的"业界共识"
- 插件对自己的局限保持诚实

如果发现它自信地说错了——或者置信度等级和实际不符——[提个 issue](https://github.com/Hayes-Zhang/best-practices/issues)。
这是最有价值的贡献。

## 要求

### 自动模式（对话中默认行为）

- 支持 plugin 的 **Claude Code**
- 可用的 **WebSearch / WebFetch** 工具
- 可选：**Context7 MCP** 用于查询框架一手文档

### `/bp` 深度模式

- 以上全部，加上：
- 环境中可用的 **Agent Team API** 工具：`TeamCreate`、`TeamDelete`、
  `TaskCreate`、`TaskUpdate`、`TaskGet`、`TaskList`、`SendMessage`，
  以及支持 `team_name` + `name` 参数的 `Agent` 工具。
  这些是启动 3 agent（scout/analyst/creator）并行调研团队的前提
- 推荐用 **Opus 模型**跑 agents（3 个并发实例——这是 `/bp` 最主要的成本来源）
- 充足的 token 预算：一次 `/bp` 大约消耗**普通对话 3-5 倍的 token**

> **注意**：如果你的环境不支持 Agent Team API，`/bp` 将无法启动团队。
> 自动模式可以正常使用。

## 局限性

- 置信度分级是启发式的，不是统计意义上的概率。模型的判断决定了上限
- `/bp` 消耗约普通对话 3–5 倍的 token（3 个 Opus agent 并行）。谨慎使用
- 自动搜索会让被触发的回答多 10–30 秒延迟。第一句"Let me look into..."
  就是为了让你知道它在干活，不是卡死了
- 来源知识在技术和产品领域最丰富。其他领域（生活技能、育儿、小众爱好）同样
  好用——只是会看到不同比例的 🔧 Synthesized、🧩 Adapted 和 💡 Deduced 标签，
  因为不同领域的资料分布不同，不是因为回答质量更低。🧩 Adapted 和 💡 Deduced
  都是正式输出层级，不是"做得不好"的信号——它们是插件诚实处理"没有直接答案"
  这个事实的方式
