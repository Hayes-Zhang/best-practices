# Best Practices for Claude Code

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
[![Version](https://img.shields.io/badge/version-2.1.0-blue.svg)](CHANGELOG.md)
[![Claude Code](https://img.shields.io/badge/Claude_Code-Plugin-orange)](https://docs.claude.com/en/docs/claude-code/plugins)

**给 Claude Code 加一层求知谦逊。** 当你问到你不熟悉的问题——技术的也好、
非技术的也好——AI 会先停下来，去查有没有人已经想过/做过/总结过，评估
找到的资料质量，再带着置信度回答。如果找不到可靠的现成方案，它会基于
第一性原理自己推导一套，并明确标注是「原创推导」而不是「业界共识」。

## 为什么需要它

AI 会用训练数据 confidently 回答你——这就是问题所在。

- 它**不知道自己不知道**
- 它**不会意识到这件事可能别人已经想过**
- 它**不会主动去找反面证据**
- 训练数据浅或者过时的时候，它**仍然会给你答案**——只是没那么对而已

简单粗暴的"每次都搜网"也有自己的失败模式：SEO 文章、AI 灌水、社区互相矛盾的
帖子，会以更"权威"的样子把模型带偏到另一个错方向。

这个插件夹在两个失败模式中间。当你问到你不熟悉的问题，AI 会：

1. **搜索**这个话题已有的最佳实践
2. **评估**找到的资料质量（结构化反向思考）
3. **标置信度**让你知道什么时候可以信
4. **找不到时基于第一性原理推导**——不是拒绝、不是瞎猜，是诚实地说"我自己想了一套"

适用于所有 *"我该怎么做 X？"* 类问题——**不限于技术**。

## 不只是技术问题

这是大多数人没意识到的——它不仅好用在：

> *"Next.js 15 怎么做认证？"*
> *"PostgreSQL 高写入场景的索引怎么设计？"*

也同样好用在：

> *"刚带的新下属，怎么开第一次 1:1？"*
> *"创业公司谈薪酬最好怎么聊？"*
> *"PR description 怎么写别人才愿意看？"*
> *"挑食的小孩吃饭怎么处理，不要每天吃饭都变战斗？"*
> *"远程团队的异步 standup 怎么开比较合理？"*

模式很简单：**只要这件事别人已经想过、而你脑子里还没有这个答案**，就适用。

## 它做什么

```

│ 自动模式（默认）                                              │
│                                                             │

│ → 跑反向思考质量评估                                          │
│ → 标置信度（✅ 找到 / 🔧 综合 / 💡 推导）                     │
│ → 把答案融入正常回复，附来源和 trade-off                       │
│                                                             │
│ 如果找不到可靠来源：基于第一性原理推导一套方案，                │
│ 并明确标注为「推导」。                                         │
│                                                             │
│ 时间预算：约 30 秒。不需要任何命令。                           │
└─────────────────────────────────────────────────────────────┘
                          ↓
              （需要更深时升级）
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ 深度模式 — /bp "主题"                                        │
│                                                             │
│ 3 个 agent（scout/analyst/creator）并行调研，                │
│ Hub-and-Spoke 协作，多来源交叉验证，                          │
│ 保存为 Markdown 报告。                                       │
│                                                             │
│ 时间预算：5–10 分钟。手动触发。                               │
└─────────────────────────────────────────────────────────────┘
```

## 快速开始

### 安装

```bash
# 添加 marketplace
/plugin marketplace add Hayes-Zhang/best-practices

# 安装插件
/plugin install best-practices@best-practices
```

### 试一下

在新对话里问任何 how-to 问题。下面这些都应该触发：

> *"Next.js 15 怎么做认证？"*（技术）
> *"团队的 oncall 排班一般怎么安排比较合理？"*（流程）
> *"冷启动邮件怎么写对方才会回？"*（沟通）

你会看到回答末尾带其中一个标签：
- **✅ Found（找到）** — 多个来源一致，这是大家的共识
- **🔧 Synthesized（综合）** — 从多个碎片来源综合的方案
- **💡 Deduced（推导）** — 没人系统总结过，这是基于第一性原理推出来的

需要更深入的多角度调研：

```
/bp "你的话题，技术的或非技术的都行"
```

更多真实输出见 **[examples/](examples/)**。

## 触发清单

插件设计为自动激活，但只在真的有价值时触发。

| ✅ 触发 | ❌ 不触发 |
|---|---|
| how-to 类问题（*"怎么做 X？"*） | 已经在执行（*"帮我把这段翻译成中文"*） |
| 选型对比（*"X vs Y"*） | 基础概念（*"什么是闭包"*） |
| 明示时效（*"最新的"*、*"2026 的"*） | 数学 / 经典算法 |
| 明示版本号（*"React 19"*、*"Tailwind v4"*） | 修具体代码 bug |
| 变化快领域（AI、前端、云服务、设计系统） | 纯创意任务（写诗、起名字） |
| 用户不熟悉的领域 | 用户已经选定方案 |
| 明确关键词（*"最佳实践"*、*"推荐"*） | 元对话（*"你能做什么"*） |
| 非技术 how-to（1:1、招聘、育儿、做菜等） | |

**多搜 30 秒的代价很小。confidently 给一个错答案的代价很大。不确定就搜。**

## 工作原理

### 三层判断

1. **触发检测**——这个问题值得搜吗？
2. **反向思考评估**（默默进行，回答前完成）：
   - 这些来源真的独立吗？还是都引用同一个原始出处？
   - 我搜过反面证据吗？
   - 这些建议有 trade-off 吗？还是所有人都在吹？
   - 信息有多新？这个领域变化多快？
3. **分级回答**：
   - **✅ Found** — 多个权威来源一致 → 直接给建议
   - **🔧 Synthesized** — 碎片化共识 → 给建议但标注"综合多方"
   - **💡 Deduced** — 找不到可靠现成方案 → 基于第一性原理推导 + 推导逻辑 + 明确标注

### 五个红旗

插件主动检测并明示：

- 🚩 **利益冲突**——来源在推销自己的产品
- 🚩 **SEO 文章特征**——标题党、内容空泛、没有具体示例
- 🚩 **AI 生成嫌疑**——高度通用、缺少具体经验
- 🚩 **银弹思维**——复杂问题只给一个方案，没有 trade-off
- 🚩 **独立性假象**——多个来源其实都引用同一个原始出处

## Trust & Safety

整个插件围绕一个原则：**诚实的来源标注 > 自信的瞎猜，诚实的第一性原理推导 >
自信的瞎猜——但前提是清晰标注。**

- 每个引用的 URL 都是真实的。URL 不存在就不编造
- 找不到可靠来源时，插件**不会拒绝回答**——它会切到第一性原理推导模式，
  展示推导逻辑，并明确标注答案是"推导"而非"找到"
- AI 推导的建议绝不会伪装成"业界共识"
- 插件对自己的局限保持诚实

如果发现它自信地说错了——或者置信度等级和实际不符——[提个 issue](https://github.com/Hayes-Zhang/best-practices/issues)。
这是最有价值的贡献。

## 要求

### 自动模式（对话中默认行为）

- 支持 plugin 的 **Claude Code**
- 可用的 **WebSearch / WebFetch** 工具
- 可选：**Context7 MCP** 用于查询框架一手文档

### `/bp` 深度模式

- 以上全部，加上：
- 环境中可用的 **Agent Team API** 工具：`TeamCreate`、`TeamDelete`、
  `TaskCreate`、`TaskUpdate`、`TaskGet`、`TaskList`、`SendMessage`，
  以及支持 `team_name` + `name` 参数的 `Agent` 工具。
  这些是启动 3 agent（scout/analyst/creator）并行调研团队的前提
- 推荐用 **Opus 模型**跑 agents（3 个并发实例——这是 `/bp` 最主要的成本来源）
- 充足的 token 预算：一次 `/bp` 大约消耗**普通对话 3-5 倍的 token**

> **注意**：如果你的环境不支持 Agent Team API，`/bp` 将无法启动团队。
> 自动模式可以正常使用。

## 局限性

- 置信度分级是启发式的，不是统计意义上的概率。模型的判断决定了上限
- `/bp` 消耗约普通对话 3–5 倍的 token（3 个 Opus agent 并行）。谨慎使用
- 自动搜索会让被触发的回答多 10–30 秒延迟。第一句"Let me look into..."
  就是为了让你知道它在干活，不是卡死了
- 来源知识在技术和产品领域最丰富。其他领域（生活技能、育儿、小众爱好）同样
  好用——只是会看到不同比例的 🔧 和 💡 标签，因为不同领域的资料分布不同，
  不是因为回答质量更低。💡 Deduced 是正式输出层级，不是"做得不好"的信号
